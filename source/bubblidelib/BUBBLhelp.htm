How to play sound

Index

  1. Right-click on the desktop and select 'Add Instruction' from the menu
    Move the mouse over 'Output` and select 'Play Sound' from the pop-up menu.

  2. Drag the new instruction to where you want it in your program and right-click on it to give it the filename or URL of the `clip` (Eg '.WAV' or '.AIF' file) you want to play.

  3. Note: Each clip playing has an entry in the system table 'CLIP'. Clips can be paused, resumed and started at any position by updating the appropriate fields in the table. Setting 'Playing' to 1 starts playing the clip from 'Position', writing 0 to it pauses the clip. The fields 'Duration' and 'Position' are in units of milliseconds (ms).
    A clip can be played without a 'Play sound' instruction by inserting a record in 'CLIP' with 'Name' equal to the file name (or URL) of the clip. Deleting a record from CLIP stops a clip playing immediately.
    When a clip is started with a 'Play sound' instruction, any clips which have run to the end are removed from 'CLIP'.
    When a clip plays to the end a 'CLIP_END' event is generated. This allows a program (for example) to loop a clip repetitively, or play a sequence of clips in a controlled manner.

  4. Note: MP3 files are not currently supported by BUBBL. If you want to play MP3 files, they should first be converted to '.WAV' or '.AIF' format.

A BUBBL program is a diagram made up of interlinked instructions. The language itself has about 30 such instructions which cover all the things which a programmer may tell the computer to do. As a programmer, you can also create your own instructions (or 'blocks') which can be used in programs in just the same way as the built-in ones.
All available instructions are shown by right clicking on the desktop and moving the mouse over 'add instruction'. The programmer-created instruction blocks are shown by moving the mouse over 'Library'. To add an instruction to your program, simply press the mouse on it and drag it onto the desktop or window. Then you can provide any extra information the instruction needs by right-clicking on it. This usually brings up a form to fill in.
The computer runs a program by `executing` (i.e. doing) an instruction, then following one of the instruction`s `links` to another instruction. It then executes the linked instruction in the same way, following links and executing instructions until it has nothing left to do (either it gets to an `End` instruction, the user clicks on the STOP button, or it gets to an unconnected link).
To start a program running double-click on the instruction you want it to start with. This 'activates' the program, (you should see a blinking rectangle around the instruction). Next click the `Run` button or, if you are being cautious, click the 'Step' button so you can check the results of the instruction and correct any mistakes before they cause problems later. If you need to do a correction you can also use the 'Back' button to 'undo' the last instruction, do the correction, then continue testing without having to go back to the beginning.

  Note: When stepping through a programs, Instruction-blocks can be 'stepped-into' or 'stepped-over' (i.e. treated as a single instruction) depending on whether the block' editor window is open.

To make program development and testing easy, you can have formulas, calculations, tables and other 'live' information visible in your program. These are called 'watches' and can be added by selecting 'Watch' from the desktop right-click menu.
You can also annotate your program with text and/or graphics by left-clicking in a blank area, or selecting 'Graphics' from the desktop right-click menu.

More Debugging tools

The system variable DEBUG when written to, causes the value written to be echoed to standard output (e.g. in the terminal window if that is where Java was run from). This may be useful for debugging things in real-time which happen too fast to see otherwise.
Also, assigning a file path and name to the variable "UNDO_DUMP_FILE" will cause the 'undo script' of the last few thousand instructions executed to be output to the specified file. Interpreting this file is beyond the scope of this document, however, the reader is encouraged to discover this for his/herself by careful experimentation!

What is it?
1. A classroom tool for teaching/learning concepts in computer science
2. A simple to use, fast to deployment, programming package for Windows/Linux etc..
3. A literacy for human/computer interaction using everyday information representation paradigms.
4. A serious 'toy' to allow 'free-play' with information, ideas, and computation.

Philosophy

At the heart of the BUBBL philosophy is the idea that everything can be represented by a string, and that there should be no 'hidden meanings' in source code. This leads to context being used to distinguish data types, rather than 'invisible' namespaces; the use of well-defined operators to compute functions (rather than mathematical function notation); and to the lack of a need for special syntax-related characters like  { , }  and  ; .  BUBBL is designed to map more naturally onto how human beings think, rather than how computers process.

The second essential feature of BUBBL is the way it deals with it's 'world'-i.e. inputs, outputs, storage and real time. Inputs, outputs, storage, and important pieces of system information (e.g. clipboard) are concrete elements of BUBBL, and real-time is encapsulated with the behaviour diagrams.  A BUBBL behaviour diagram describes completely how the computer (or, more precisely, the BUBBL virtual machine) should behave. Algorithms are precisely represented with its flow-chart structure, and 'states' of the program are precisely represented with its state-transition structure. There is no need for 'locks', 'synchronisation blocks' and 'event listeners' as these are handled in a well-defined way by the BUBBL vm.

The third cornerstone of BUBBL is the full integration of comments and annotations (e.g. including images, highlighters and free-hand drawing) into the 'source code'. There is no run-time overhead or penalty for inclusion of these elements, as the BUBBL VM loader compiles only the working instructions as it loads them.

The final key philosophical dimension to BUBBL is the lack of an 'error-trapping' or 'exception handling' mechanism. Because functional elements of BUBBL (blocks) can have more than one exit link, there is no need for a special mechanism to handle 'errors'. 'Exceptions' are simply handled by giving them a link. What is more, it is easy for the programmer to deal with exceptions, as the links appear in the code in the same place as where decisions should be made about them!. There can be syntax errors or 'runtime' errors (e.g. division by zero) in expressions, but these result in the expression evaluation to a string representation of itself. When debugging a program (and such errors are always bugs), it is easy to see where the errors occur by stepping forwards or backwards through the program whilst watching variables and expressions.

From an object-oriented perspective, classes and objects are database entities - simple named relations and tuples respectively.
Methods are operations deriving from and/or on tuples and sets of tuples (relations).
In the case of (e.g) the DISPLAY table, the database entities are mapped onto the 'objects' displayed through reading and writing table values.

File system instruction

Index

There are 9 types (sub-functions) of file system instruction. Each sub-function may succeed or fail, and the corresponding link is followed. If it fails, the system variable 'FILE_MESSAGE' is set to a message describing the reason for the failure, other wise it is set to "Ok". The sub-functions are as follows:-

alpha channel

Images with an alpha channel allow a 'transparency' effect. Each pixel of the image has an alpha value in the range 0 to 255, with 0 indicating totally transparent and 255 being totally opaque. When the images are painted onto a window the alpha value is used to calculate how much of the colour already present 'shows through' to the pixel being drawn.

annotation

BUBBL provides a way to store additional data about tables, these additional data are called 'annotations' and can be accessed in two ways: either from the column 'annotations' in the table 'TABLE', or the column 'value' in the table 'TABLE_METADATA'.

assign

When a BUBBL program stores a value in a variable, this is called 'assigning' a value to the variable. It is similar to the idea of 'let' in formal algebra: e.g. "let x=5" assigns the value 5 to the variable 'x'. The BUBBL 'Let' (or 'Assign') instruction is used to assign values to variables.

column

Part of the contents of a table, consisting of the set of all row values for a particular column of the table.
'column' is synonymous with 'field'

data

Information represented as sequences of characters. E.g. '23' is a representation of the number 23.

database One or more 'tables' which contain related information.
design pattern A formal description of a solution to a common 'problem' (or system requirement) in software development which is generally held to be the 'best practice' for solving the 'problem'.
event A real time occurrence of something relevant, such as a key-press or mouse movement. In BUBBL there are built in events for keyboard and mouse input, timing and other uses
execute 'Run' or 'enact' a program or instruction.

exit link

 When programming instruction blocks, it is possible to create multiple links from the instruction block (by editing the start instruction). Within the instruction block each 'end' instruction has its own connection to one of the links. The links are called 'exit links'. They can (and should!) be labelled.

expression

A combination of one or more values and/or variable names and operators as a sequence of characters. Expressions can be 'evaluated', meaning that all operators are applied in order of precedence until a constant value remains – the 'returned value'

field

Refers to a column of a table, or part of a 'tuple' variable or table row. In tables, fields are referred to by name, or by number (1 refers to the leftmost). In simple tuple variables they are referred to by number, e.g. “one,two,three”.1 is “one”

generator

A design pattern in which iterations are 'generated' from a block of code rather than a formal looping structure of the language. It allows reusable 'templates' to be written, where the implementation details of some general-purpose code are delegated to code not in the 'generator' block. In BUBBL generators can be implemented using a link for each delegated task and the system variable 'NXT' to test whether a block is entered normally, or via a 'Next' instruction, indicating returning from a delegate.

instruction

One of about 25 codified actions that the computer can do. They can be joined together via various 'links' to make a network which represents the programmed behaviour.

instruction-block

An instruction made up a network of other instructions. An instruction block always has a single 'Start' instruction and may have zero or more 'End' instructions.

iterate

 This means to repeat the same thing over and over. The For/Next instruction is used to repeat one or more instructions for each of a set of integers.

link

 The connection between two instructions. Different links starting from the same instruction or block are usually labelled, and represent different decisions of the instruction or block.

looping

 Looping is a form of iteration, where the same thing is done repeatedly, perhaps using different data each time.
Loops typically have some kind of test which allows the loop to 'exit'.
With the 'C' programming language and others, loops are constructed with 'while', 'do while' and 'for' statements with 'continue' and 'break' statements used to 'break out' of or 'short-circuit' a loop.
In BUBBL loops are made either by linking instructions into a loop, with 'if' and/or other multiple-link instructions used to exit, short-circuit or break out of the loop, or by using a 'for-next' pair of instructions where the 'looped' instructions connect the 'Do' link of the 'For' instruction to the 'Next' instruction.
mark-ups information passed to Output instructions to affect screen position, font, colours etc.
output Information produced by a program for the 'user' to see, read or hear. Examples include writing text, showing an image and playing a sound clip.
page A 'window' in which a program's 'output' may be displayed.

parameter

Parameters provide an instruction or instruction block with the detail it needs to work.
Parameters may be 'input only' or 'input/output'.
Input only parameters provide information to the instruction or block.
Output parameters get information from the instruction or block.
Input/Output parameters bring information in, but they also carry information out of the instruction or block. An instruction block's start instruction lists all of the block's parameters.

program A network of instructions which describes how the computer should behave. A program can be 'executed' (or 'run'). The diagram represents 'states' which determine behaviour in reponse to 'events'.
record Another name for a row of a table, or single data item with many named 'fields'

relation

 In mathematics this is a formal way of representing the relationships between items of information or data. A relation can be thought of as a table with named columns and numbered rows. The name of the table is the name of the relation, and the column titles are the 'fields' of the relation. Each column, in principle, has some sort of relationship to other columns, and all have a relationship to the 'name' of the relation.
E.g. A simple relation used to express personal details:
    person(forename,surname,date_of_birth,home_phone)
and some members of the relation (or 'tuples'):
    person(John,Smith,1-12-1990,123456)
    person(Julie,Snobbit,23-4-1984,102032)

row

The primary building block of a table. It consists one value for each column (field) of the table. The values are separated by commas. Rows may also be referred to as 'tuples'.
Record 0 of a BUBBL table contains a row whose field values are the field names. Note: When handling a row in its entirety, if a column value contains one or more commas they are instead expressed as '|' to prevent them being confused with the row's field boundaries. They are, however, expressed as commas if the field is referred to explicitly.

set

 In BUBBL sets are made up of integers. They can usefully be used to index multiple rows of tables and also to handle ranges. Sets are surrounded by braces (curly brackets {})
E.g. {1..5,14,34} has 7 elements, 1,2,3,4,5,14 and 34

The Display

Index

The idea of a 'page' is used to organise visual output. Pages can be created and destroyed using `Page-Select` and `Page- Destroy` instructions, and can be printed and dumped to image files.

Some output is subject to 'mark-ups' which can be used to control things like fonts and colours (see Output mark-ups ).

The display management system maintains some variables: CP, CX, CY, INK, PAPER, FONT, ITALIC, BOLD, PX, PY, PW, PH, DW DH, SW, SH. Most of these variables apply to the current page (CP) only. In general they cannot be directly assigned to, but may change as a side effect of some other action such as use of the 'Page-Format' instruction. For example, `CX` and `CY` are modified by the 'Ouput-Expression' and 'Output @' instructions.

Normally the display is maintained using the 'Output' and 'Page` instructions. The standard unit of size is the 'Point' (1/72 of an inch for printing). All screen dimensions and coordinates etc. follow this convention. Additionally everything in the contents of the current page is availble to the program through the table variable `DISPLAY`.

Outputing to pages is affected by the current `mode`: If in `direct-write` mode any program output is displayed immediatley on the page, and anything written in this mode is subject to on-screen corruption by other screen objects (for example other software running on the computer).

Normally however,a list of display items is maintained automatically, and the items persist until expressly overwritten, scrolled or cleared off the screen. A page can also be 'frozen' so that items written to it before the 'Page-Freeze' instruction are not removed or scrolled, creating a static 'background'.

Items added to the display list after any `Page-Freeze on` instruction, or all display items, are subject to the following rules for removal from the display list.

  1. New items are appended to the end of the table (as in table insert instruction).
  2. 'Text' items, 'line' items (lines, rectangle, circle and polygon outlines and sectors) and 'area' items (solid rectangles, pictures etc.) are treated differently.
  3. Subject to the behaviour described above, an item will be removed if:
    1. A 'clear screen' instruction is executed
    2. The list has>2000 items, and the item is number 1
    3. It's y-coordinate is -Ve after scrolling
    4. It is completely overlapped by a new 'area' object
    5. It is a text item and a new text item has the same co-ords
However, an item may be displayed (appended to the table) without affecting others at all if the 'a' (for append) mark-up is used.
